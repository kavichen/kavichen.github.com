简介

Divide objects in your program into 3 "camps".

  1. Model
  2. Controller
  3. View

Model

  • Model = What your application is (but not how it is displayed)
  • The Model and View should never speak to each other.
    • Model is completely UI independent.
  • The data source in the delegate always is the controller, or the third class designated by controller, but they will never be the model.
  • So what if the Model has information to update or something?
    • Ans: It uses a "radio station"-like broadcast mechanism. Controllers (or other Model) "tune in" to interesting stuff. A View might "tune in", but probably not to a Model's "station".

Controller

  • Controller = How your Model is presented to the user (UI logic)
  • Controllers can always talk directly to their Model.
  • Controllers can also talk directly to their View. (With outlets)
  • The Controller sets itself as the View's delegate
    • will
    • did
    • should
  • Controllers are almost always that data source (not Model!).
  • The data source in the delegate always is the controller, or the third class designated by controller, but they will never be the model.
  • Controllers interpret/format Model information for the View.

View

  • View = Your Controller's minions
  • Be as generic as possible
  • The Model and View should never speak to each other.
  • View could talk to controller with Target - Action.
  • Sometimes the view needs to synchronize with the Controller.
  • Views do not own the data they display

MVCs working together

Q:Every model and every model has one controller. A:Yes.

Q:Can controller talk each other's views? A:Usually a controller would only have a point to another controller as it's view, it will ask that controller display something for it.
